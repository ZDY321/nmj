# 数据容量控制与历史归档计划

## 背景

当前系统会长期保存学生档案、课程档案、课节明细、工资核对数据和教务核对记录。学生归档本身占用很小，不是主要容量风险；后续多人一起使用后，更需要控制的是多年课节明细、回收站、长备注和单用户加密 vault 总大小。

教务核对的保存现场目前最多 6 个，空间风险已受控；课程映射长期保留且体积较小，暂不需要按月或按年清理。

## 保留原则

1. 学生档案和课程档案默认归档，不直接删除，以保证历史课节、工资核对和课程映射可追溯。
2. 近期数据保留完整明细，方便查课、核对工资、处理补课和纠错。
3. 更早年份可转为年度汇总，减少完整课节明细长期堆积。
4. 删除类数据先进入回收站，超过保留期后再永久清理。
5. 多用户场景下需要能看到每个用户的数据体积和主要占用来源。

## 后续优先级

### P1：容量统计

目标：先看清楚数据增长来自哪里，再决定清理策略。

建议统计项：

- 当前用户 vault 加密前 JSON 大小估算
- 学生数量：在读、过渡期、已归档
- 课程数量：启用、暂停
- 课节数量：按年份、按状态统计
- 回收站课节数量和估算体积
- 保存的教务核对现场数量
- 课程映射数量
- 备注/内容较长的课节数量

入口建议：管理后台或档案信息里的“存储管理”。

### P2：回收站清理

目标：避免已删除课节长期占用空间。

建议规则：

- 默认保留最近 90 天删除记录。
- 提供“清理 90 天前回收站”按钮。
- 清理前显示将删除的课节数量，并要求确认。
- 管理后台可查看每个用户回收站占用。

### P3：旧年度只保留工资汇总

目标：对很早年份减少完整课节明细，只保留工资核对需要的汇总结果。

建议做法：

- 以自然年为单位，例如 2024 年及以前。
- 归档前先生成年度工资汇总快照。
- 汇总快照至少包含：月份、校区、课程类型、课节数、课时数、学生人数口径、应发金额、义务课扣减、调整项、最终金额。
- 可选保留：课程名称汇总、学生名称汇总、异常备注摘要。
- 归档后完整课节明细可以删除或压缩为只读摘要。
- 操作必须二次确认，并提示“删除完整明细后不能再逐节核对”。

### P4：旧年度课节压缩/导出

目标：在删除完整明细前给用户一个可离线保存的出口。

可选方案：

- 导出年度 Excel 或 JSON。
- 导出后再执行“只保留工资汇总”。
- 系统内保留导出时间、年份、课节数量、汇总金额，方便追溯。

### P5：多人使用容量预警

目标：多人使用时提前发现异常账户或异常增长。

建议规则：

- 单用户 vault 超过预设大小时显示提醒。
- 管理后台按用户列出 vault 大小、课节数、回收站数、最近保存时间。
- 对异常大的用户提供建议操作：清理回收站、归档旧年度、导出旧数据。

## 暂不处理

- 不需要限制学生归档数量。
- 不需要删除课程映射。
- 不需要额外清理最多 6 个的教务核对保存现场。
- 不建议自动删除历史课节，必须由用户手动确认。

## 与当前功能的关系

“在读 / 过渡期 / 已归档”学生状态主要解决档案展示和业务流程问题，不是容量控制方案。容量控制应单独做“存储管理”和“旧年度归档”，避免影响当前工资核对、排课和教务核对流程。
